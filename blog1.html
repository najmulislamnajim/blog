How the Linux Kernel Manages Memory

✍️ Najmul Islam February 15, 2025 ⏱ 8 min read

Memory management is one of the most fundamental responsibilities of any operating system. The Linux kernel has a sophisticated, layered approach to handling memory — balancing performance, flexibility, and safety. In this post, we'll walk through the key concepts from virtual memory to the buddy allocator.

Virtual Memory: The Abstraction Layer

Every process in Linux operates in its own virtual address space. This is a powerful abstraction — from the process's perspective, it has access to the entire address space (e.g., 0 to 2⁴⁸ on a 64-bit system), even though physical RAM is shared across all processes.

Virtual memory decouples a process's logical view of memory from the physical hardware, enabling isolation, security, and efficient sharing.

The kernel maps virtual addresses to physical addresses using page tables. On x86-64, this is a four-level (or five-level on newer hardware) page table hierarchy: PGD → PUD → PMD → PTE.

Page Tables and the MMU

The Memory Management Unit (MMU) is a hardware component that performs virtual-to-physical address translation. When a process accesses memory, the CPU walks the page table to find the corresponding physical page frame.

/* Simplified page table walk concept */
unsigned long virt_to_phys(struct mm_struct *mm, unsigned long vaddr) {
    pgd_t *pgd = pgd_offset(mm, vaddr);  // Level 1
    p4d_t *p4d = p4d_offset(pgd, vaddr); // Level 2
    pud_t *pud = pud_offset(p4d, vaddr); // Level 3
    pmd_t *pmd = pmd_offset(pud, vaddr); // Level 4
    pte_t *pte = pte_offset_kernel(pmd, vaddr); // Level 5

    return (pte_val(*pte) & PAGE_MASK) | (vaddr & ~PAGE_MASK);
}

To avoid this expensive walk on every memory access, the CPU caches recent translations in the Translation Lookaside Buffer (TLB). A TLB miss triggers the full page table walk.

Page Faults

When a process accesses a virtual address that isn't currently mapped to a physical page, the CPU raises a page fault exception. The kernel's page fault handler then decides what to do:

  • Minor fault — the page is in memory but not mapped (e.g., demand paging). The kernel maps it and resumes.
  • Major fault — the page is on disk (swapped out). The kernel reads it back from swap, maps it, and resumes.
  • Segfault — invalid access. The kernel sends SIGSEGV to the process.

Physical Memory Management: The Buddy Allocator

For managing physical page frames, Linux uses the Buddy Allocator. Physical memory is divided into blocks of sizes that are powers of 2 (1, 2, 4, 8, … pages). When an allocation is requested, the allocator finds the smallest fitting block.

# View buddy allocator state:
cat /proc/buddyinfo

# Example output:
# Node 0, zone  DMA32    2  3  4  2  3  1  2  1  2  1  4
# Each number is the count of free blocks of 2^n pages

When a block is freed, if its "buddy" (adjacent block of the same size) is also free, they are merged into a larger block. This prevents external fragmentation at the page level.

Slab Allocator

For kernel objects smaller than a page (like task_struct, inode, etc.), Linux uses the Slab/SLUB Allocator. It maintains caches of pre-allocated objects to avoid the overhead of repeatedly allocating and freeing small structures.

# View slab cache info:
sudo cat /proc/slabinfo | head -20

# Or with slabtop:
sudo slabtop

OOM Killer

When the system runs out of memory and cannot free any more, the kernel's OOM (Out-of-Memory) Killer is invoked. It scores processes based on their memory usage, runtime, and niceness, then kills the process with the highest score to reclaim memory.

Conclusion

Linux memory management is a beautifully engineered system. Virtual memory provides process isolation, page tables enable address translation, and allocators (buddy + slab) handle physical memory efficiently at different granularities. Understanding these internals helps you write more memory-efficient code and debug tricky issues like memory leaks or OOM kills.
